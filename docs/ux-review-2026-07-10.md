# UX review — live-site walkthrough, 2026-07-10

Cold-visitor walkthrough of wyreup.com (desktop Chrome, production). Journeys
exercised: landing comprehension → tool discovery → run Compress with a real
file → chain into Convert Format → search → text tool (Word Counter) → Pro
tool gate (Upscale) → pricing → Toolbelt → chain builder. Mobile viewport not
tested (tooling limitation) — do a separate pass.

Verdict on the "are we as usable as we think?" hypothesis: **the core runner
loop is genuinely good — the problems are at the edges that shape trust and
first impressions.** Drop → params → run → result → chain-next works, is fast,
and reads clearly. What undermines it: a homepage that looks finished after
one screen, tools that can silently do the wrong thing (Compress inflating a
file 37×), machine-facing output for human questions (raw JSON word counts),
false privacy copy on hosted tools, and a flagship chain builder whose tool
picker is a 261-option native dropdown.

## P0 — trust and correctness

1. **Compress can inflate files with no warning.** A 24.7 KB flat-color PNG
   re-encoded at quality 80 came back as **943.3 KB (“Change: +3713.8%”)**,
   styled like any other result with a prominent Download button. Screenshots
   (the most common desktop “compress this” input) will hit this constantly.
   Expected behavior: when output ≥ input, keep/offer the original and say
   “already smaller than the target encoding can manage.” Likely a systemic
   pattern — the phase-1 tool review should check every optimize/convert tool
   for “output worse than input” handling.
2. **Hosted Pro tools carry the local-only privacy boilerplate.** The Upscale
   (PRO) page says “Hosted Real-ESRGAN … the GPU does the heavy lifting” and,
   three paragraphs later, “Upscale runs entirely on your device. Files you
   provide never leave your browser — no uploads, no server.” Both cannot be
   true; the privacy box is auto-stamped per tool page. On a brand whose hero
   is “Nothing uploads,” this is the single worst trust bug on the site.
   Hosted tools need an honest variant (“sent to Wyreup’s GPU backend over
   TLS…”). Audit all 30 PRO tool pages.

## P1 — first-visit conversion

3. **Homepage false floor.** The hero fills the viewport and ends in a
   footer-looking stats strip (“261 tools · 0 uploads · 0 tracking · open
   source”). It reads as the end of the page; the only scroll affordance is a
   tiny orange “scroll ⌄” at bottom-right. The excellent sections below
   (use-case cards, Drop/Pick/Run, chain demo, popular tools) risk never being
   seen.
4. **Forced 100vh section rhythm.** Every homepage section is exactly one
   viewport tall regardless of content, leaving 300–500 px of dead black
   between blocks; `End`-key lands on a fully blank screen. Fit sections to
   content.
5. **Chain-builder tool picker is a flat native `<select>` with 261 ungrouped
   options** — no search, no categories, no input/output-compatibility
   filtering at pick time. This is the flagship differentiator; it deserves
   the same searchable, categorized picker the site search already has.
6. **“Toolbelt” nav label.** First-time visitors click into an empty page
   about chains/trigger rules they haven’t met yet. Rename (“Saved chains”)
   or gate behind first save.

## P2 — persona fit (“simple for mom, capable for engineers”)

7. **Inspect-category tools return raw JSON as the human-facing result.**
   Word Counter shows `{"words": 42, …}` in a code block. Needs a readable
   stats rendering first (42 words · 4 sentences · ~12 s read), raw JSON
   behind Copy/Download. Likely applies to most `application/json`-output
   tools (text-statistics, password-strength, color-contrast…).
8. **Chain suggestions are MIME-driven, not semantics-driven.** Word-count
   output suggests Hash, Base64, CSV↔JSON, and **GeoJSON to KML** — nonsense
   hops that make the chaining feature look random. Suggestions need a
   semantic layer (or curated per-tool next-steps), not just MIME match.
9. **Pro gate omits the offer.** The “Get an API key” modal (nice: one field,
   no card) never says what a key includes or costs — no credit grant info,
   no pricing link. The strong /pro page copy (“Packs from $5 · credits never
   expire”) should be one line in the modal.

## P3 — polish

10. Chain handoff requires an extra “Use this file” confirm after already
    clicking the suggestion (defensible for preview-before-run, but consider
    pre-loading with an inline undo).
11. “Save intermediate” checkbox on results is unexplained jargon.
12. 404 page has personality but no search box or tool suggestions.
13. Search-results fuzziness: “word count” surfaces HOTP Code, Password
    Strength, Flatten PDF Form alongside the right hits.
14. Near-duplicate tool naming: “Upscale” (PRO, hosted, slug
    `upscale-2x-pro`) vs “Upscale 2x” (free, local). Confusable pair —
    naming should encode the difference (e.g. “Upscale (hosted GPU)”).

## What is already strong (don’t break in fixes)

- Result → “Use this result in …” chaining works end-to-end and is the right
  interaction; hop 2 (943 KB PNG → 52 KB WebP, −94.5%) felt great.
- Search: instant, categorized, keyboard-driven (Cmd+K), with an “email a
  suggestion” escape hatch.
- Tool pages: dropzone state feedback, param helper text, About/Common
  uses/FAQ depth, model-size (“15 MB on first use”) and PRO/AI badges.
- /pro page: clear packs, concrete credit equivalences, no-password account,
  auto-refund on failed runs.
- Homepage sections that DO exist below the fold (use-case cards, Drop/Pick/
  Run, example chains on /tools) are exactly the right content.

## Feed into phase 1 (per-tool review)

Systemic checks every tool reviewer should run: (a) output-worse-than-input
handling, (b) privacy-box truthfulness vs. execution location, (c) JSON-vs-
human result rendering, (d) sanity of the chain-suggestion list, (e) naming
collisions with sibling tools.
