## The one remaining "high" — astro server-islands XSS, not applicable

`GHSA-wrwg-2hg8-v723` — *"Astro vulnerable to reflected XSS via the
server islands feature"*. Vulnerable: `astro <= 5.15.6`; patched: `>=
5.15.8`. We're on `astro@4.x`.

**Why we're not on astro 5 yet:** the astro 5 + svelte 5 + vite 6
combination has a known build-time regression where vite's
`build-import-analysis` plugin can't parse svelte 5's SSR-generated
chunks (specifically the `_@astro-renderers.<hash>.mjs` virtual module
re-exports from `svelte/server`). Tested at @astrojs/svelte versions
7.2.0–7.2.5 with svelte 5.1 and 5.43; same failure either way. Astro 6
moves past it but drops compatibility with `@vite-pwa/astro` which is
load-bearing for the PWA install + share-target flows.

**Why the advisory doesn't apply to us:** Server islands are a feature
of astro's hybrid-SSR mode. We're configured `output: 'static'` (see
`packages/web/astro.config.mjs:7`) — pure SSG, every page pre-rendered
at build time. No server-island rendering happens at request time
because there is no request-time rendering at all. We're not exposed
to the XSS vector.

**Plan to clear the advisory anyway:** wait for one of:
- An astro 5.x patch that fixes the chunk-naming / vite import-analysis
  interop (or a vite 6 patch from the other side)
- @vite-pwa/astro to add astro 6 support (then jump to 6 + @astrojs/
  svelte 8)

Re-audit on each pnpm install. Don't ship astro upgrades blindly.
